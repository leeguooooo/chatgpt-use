chatgpt-use · 架构与研究纪要

把 ChatGPT 网页订阅接入本地 coding-agent 工作流 · 2026-06-17 · 实验性

本文记录:我们撞到的墙(实证)、社区三个成熟项目的做法、由此收敛出的目标架构(双通道), 以及决定保留的自有模式。一句话定位:把 ChatGPT(尤其 GPT-5.5 Pro)当高质量 planner/reviewer 接入本地 coding-agent 流程,执行始终留给 Codex / Claude Code / 本地工具。

1. 实证结论:浏览器里"让 ChatGPT 演工具"这堵墙

我们先按"网页版无原生 function-calling → 用文本协议伪造工具调用"的思路做了 Mode 2/3,并在真实登录态网页上实测。 跨 3 种提示词框架 全部被模型拒绝:

框架结果
"你拥有这些工具,按需调用"模型用自然语言要求"上传/粘贴文件"
"你没有任何文件访问权限,工具是唯一途径""那我无法完成,请提供文件"
"你不是在执行工具,只是输出 JSON 请求,外部控制器会执行并贴回结果" + few-shot "贴更多 schema 文本并不能让工具变真,请提供文件"
结论:强 grounding 的网页模型不会仅凭单条消息文本去"扮演工具执行"。 这不是提示词没调好,而是方向本身要换 —— 三个成熟项目无一硬破此墙。

2. 社区三个项目(源码级)

ChesterRa/cccc 930★ · 活跃

单写者 daemon + append-only ledger.jsonl 为唯一事实源;Web/CLI/MCP/IM 都是无状态前端。 对接 ChatGPT 用双通道:

  • 浏览器投递(push):daemon 持有持久化 Playwright/CDP profile,把消息批量注入绑定的 chatgpt.com/c/… 会话;提交后若无法确认接收则标 delivery-unverified。
  • Remote MCP(pull):暴露公网 HTTPS MCP 端点(需 Cloudflare Tunnel / ngrok / Tailscale, 因为 ChatGPT 在云端、够不到 localhost);ChatGPT 注册为自定义 app connector,GPT-5.x 调 cccc_runtime_wait_next_turn 拉取轮次、用 cccc_repo_edit/cccc_shell/cccc_git 等工具。 URL 带 actor-bound token,服务端只存哈希。
Pro 硬约束(cccc 文档原文):"GPT-5.x Pro sessions currently cannot be used this way — they do not expose third-party MCP connectors." Pro 只能当"需手动给上下文的外部顾问"。

tt-a1i/hive 369★

纯本地 CLI PTY 工作台,不接 ChatGPT 网页。值得借鉴:把 <xml-system-reminder> 锚点放在每条消息尾部(抗 /compact 与自动摘要)、 PROTOCOL.md 兜底锚点、tasks.md 任务图、通过读 rollout-*.jsonl 首行恢复会话。

RPG-478/codex-chatgpt-bridge alpha · 最对口

本地 cgpt CLI + stdio MCP server,让 Codex 把小型结构化"委派包"丢给 ChatGPT 网页(纯 Playwright)。 ChatGPT 只规划/评审,执行留在 Codex。其浏览器收发逻辑与我们 channel.rs 几乎一致(填编辑器→点 [data-testid=send-button]→轮询 [data-message-author-role=assistant]→连续 2 次稳定且无 stop-button)。

委派包要点:① 委派声明(发件人是机器,不是人类)防止模型当聊天;② 模式分型 plan/review/debug/research;③ 严格输出 schema 含 verdict: proceed|revise|blocked, 解析失败直接失败不存歧义结果。

3. 目标架构:按模型分流的双通道

                       chatgpt-use (本地 daemon)
                                │
     ┌──────────────────────────┼───────────────────────────┐
 浏览器通道(chrome-use)     MCP 通道(本地 server+公网隧道)   执行 handoff
     ▼                          ▼                             ▼
 GPT-5.5 Pro              GPT-5.5 Instant/Thinking       Codex / Claude Code
 规划 / 评审               原生工具调用(无拒演墙)         本地落地执行
 (Pro 无 Apps/MCP,
  只能走浏览器)
模型第三方 App / MCP在本架构里的角色
GPT-5.5 Instant可用MCP 通道,原生工具
GPT-5.5 Thinking可用MCP 通道,原生工具
GPT-5.5 Pro不可用浏览器通道,planner/reviewer 出委派包

planner → executor 委派包(契约)

{
  "goal": "implement X",
  "plan": [ {"step":1,"action":"inspect","target":"src/foo.ts","success_criteria":"…"} ],
  "risks": ["migration may break auth"],
  "tests": ["cargo test"],
  "acceptance": ["feature works","no regression"],
  "do_not_do": ["broad refactor","change public API"],
  "verdict": "proceed | revise | blocked"
}

executor 只吃这种包、不接受 Pro 的自由文本直接执行 —— 显著降低幻觉误操作。

4. 借鉴清单 → 主线工作

借鉴自我们要做
codex-chatgpt-bridge结构化委派:ask 升级为 --mode plan|review|debug|research + 本地上下文采集 + verdict schema + fail-fast
(Pro 约束)--model 开关:浏览器通道选 Pro/Thinking(新版 UI 选择器目前自动化点不动,待解)
ccccMCP 通道(本地 server + 公网隧道,普通 5.5 原生工具);append-only ledger + actor-bound token
hive尾部 <xml-system-reminder> 锚点 + PROTOCOL.md 兜底 + tasks.md
—executor handoff:把委派包喂给 Codex / Claude Code 执行

5. 保留的自有模式(不放弃,慢慢研究)

  • Mode 1 · ask —— 已实测可用,将升级为结构化委派,成为主线。
  • Mode 2 · run(浏览器自主工具循环)—— 撞拒演墙,作为研究track,后续试真·多轮对话预置。
  • Mode 3 · serve(Anthropic drop-in)—— 同墙,暂为 PoC。

6. 合规边界

OpenAI Terms 禁止"程序化提取 Output"与绕过速率限制。本工具不定位为"网页 ChatGPT 当免费 API", 而是个人生产力桥:用你自己登录的 ChatGPT(尤其 Pro)当 planner/reviewer,执行交给 Codex/Claude Code/本地工具。 不做转售、不做规模化自动化。价值在于"订阅额度与 API/Codex 额度分桶"的合理利用。
