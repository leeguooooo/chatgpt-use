MCP 通道接入说明

把本地项目工具暴露给 ChatGPT(普通 GPT-5.5)做原生工具调用 · chatgpt-use mcp 子命令

chatgpt-use mcp 起一个本地 JSON-RPC 2.0 MCP server,把项目工具 (read_file / write_file / list_dir / grep / bash) 暴露成 ChatGPT 的自定义 connector。普通 GPT-5.5 能原生调用这些工具——绕开"浏览器里拒绝扮演工具"那道墙。

两条硬前提,先看:
  • Pro 用不了。GPT-5.5 Pro 不支持 Apps/MCP connector。本通道只对 普通 GPT-5.5(Instant / Thinking)生效;Pro 的规划能力请走浏览器通道(ask --model pro)。
  • 需要 ChatGPT 套餐支持自定义 connector(通常 Plus/Pro/Business 的 Developer mode / Apps 高级设置;按账户、地区、工作区而定)。

为什么需要隧道

ChatGPT 跑在云端,够不到你的 localhost。所以本地 server 必须经一条公网 HTTPS 隧道 (Cloudflare Tunnel / ngrok / Tailscale Funnel)暴露出去,再把公网 URL 填进 ChatGPT。

接入四步

  1. 起本地 MCP server

    --cwd 限定工具能触及的根目录,--token 是访问口令(强烈建议,见安全警告)。

    chatgpt-use mcp --port 8788 --token "$(openssl rand -hex 16)" --cwd /path/to/your/project

    启动日志(真实输出):

    [mcp] listening on http://127.0.0.1:8788
[mcp] cwd: /path/to/your/project
[mcp] auth: Bearer token required
[mcp] tunnel hint: expose with  cloudflared tunnel --url http://127.0.0.1:8788
[mcp]   then register the public URL in ChatGPT > Settings > Apps > Add custom connector

  2. 开一条公网 HTTPS 隧道

    任选其一,把 127.0.0.1:8788 暴露成 https://…:

    # Cloudflare(零账号快速隧道,给一个临时 trycloudflare.com 域名)
cloudflared tunnel --url http://127.0.0.1:8788

# 或 ngrok
ngrok http 8788

# 或 Tailscale Funnel
tailscale funnel 8788

    记下它给出的 https://<random>.trycloudflare.com 之类的公网地址。

  3. 在 ChatGPT 里注册 connector

    ChatGPT → Settings → Apps(或 Connectors / Developer mode 的 Advanced)→ Add custom connector / Create app:

    字段填什么
    MCP Server URL第 2 步的公网地址,末尾可带 / 或 /mcp(两者都接受)
    Authentication / HeaderAuthorization: Bearer <你的 token>(即第 1 步 --token 的值)

    保存后 ChatGPT 会握手(initialize + tools/list),把 5 个工具拉到工具列表。

  4. 在普通 GPT-5.5 对话里用

    新开一个 GPT-5.5(Instant / Thinking,不是 Pro)对话,经 Tools 菜单或 @<connector 名> 启用本 connector。之后让它读/改文件、跑命令,它会真的发 tools/call 给你的本地 server 执行并拿回结果。

本地自检(无需 ChatGPT)

起 server 后,用 curl 验证(以下为真实返回):

# 无 token → 401
curl -s -o /dev/null -w "%{http_code}\n" -X POST http://127.0.0.1:8788/ \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize"}'
# → 401

# initialize(带 token)
curl -s -X POST "http://127.0.0.1:8788/?token=YOUR_TOKEN" \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize"}'
# → {"id":1,"jsonrpc":"2.0","result":{"capabilities":{"tools":{}},
#     "protocolVersion":"2024-11-05","serverInfo":{"name":"chatgpt-use","version":"0.0.1"}}}

# tools/call(Bearer header)
curl -s -X POST http://127.0.0.1:8788/ -H 'Authorization: Bearer YOUR_TOKEN' \
  -d '{"jsonrpc":"2.0","id":3,"method":"tools/call",
       "params":{"name":"read_file","arguments":{"path":"Cargo.toml"}}}'
# → {"id":3,"jsonrpc":"2.0","result":{"content":[{"type":"text","text":"[package]\n…"}],"isError":false}}

协议细节

项值
传输HTTP POST / 或 POST /mcp,JSON-RPC 2.0(无需 SSE)
protocolVersion2024-11-05
方法initialize · notifications/initialized(通知,无响应体)· tools/list · tools/call
tools/call 返回{content:[{type:"text",text:…}], isError:bool}(工具失败时 isError:true)
鉴权Authorization: Bearer <token> 头 或 ?token=<token> 查询;不符 → HTTP 401 + JSON-RPC -32000
未知方法JSON-RPC -32601

⚠️ 安全警告(务必读)

本 server 里没有人类审批环节:tools/call 一律以 auto-approve 执行, 其中 bash 与 write_file 会真的在 --cwd 里跑命令、写文件。 因此:
  • 公网 URL + token = 你机器上 --cwd 范围内的 shell 权限。谁拿到它谁就能执行命令。
  • 务必设 --token(用随机长串),把 URL 和 token 当机密;别贴进聊天/截图/仓库。
  • --cwd 收窄到具体项目目录,别用 $HOME 或 /。
  • 用临时隧道,用完即停;长期暴露请加更强鉴权/网络隔离。
  • 合规:见 architecture.html 的 ToS 边界——这是个人生产力桥,非规模化自动化。

排错

现象原因 / 处理
ChatGPT 里看不到工具多半在用 Pro;换普通 GPT-5.5。Pro 不支持 MCP connector。
握手失败 / 401connector 没带 Authorization: Bearer <token>,或 token 与 --token 不一致。
连不上 / 超时必须是公网 HTTPS(ChatGPT 够不到 localhost);确认隧道在跑、URL 没过期。
工具报错 isError:true路径不在 --cwd 内 / 文件不存在 / 命令失败——看返回的 text。
